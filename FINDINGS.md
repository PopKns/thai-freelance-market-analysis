## 1. ภาพรวมตลาดฟรีแลนซ์ไทย

| ตัวชี้วัด | ค่า |
| --- | --- |
| จำนวน gig ที่วิเคราะห์ | 2,935 |
| จำนวนหมวด | 13 |
| **ราคาเริ่มต้นมัธยฐาน (median)** | **฿550** (p25 ฿300 · p75 ฿1,500 · สูงสุด ฿2,000,000) |
| gig ที่มีรีวิว ≥1 | **~9%** |
| rating ทั่วไป | ~5.0 เกือบทุก gig |

**ราคาเบ้ขวาอย่างหนัก (right-skewed)** — งานส่วนใหญ่กระจุกที่ราคาถูก แต่มีหางยาวด้านพรีเมียม
(และมี outlier ราคาปลอมสูงถึง ฿2 ล้าน) → **ค่า median คือตัวเลขเดียวที่เชื่อถือได้** ส่วนค่าเฉลี่ย (mean)
ใช้ไม่ได้เลยเพราะโดน outlier ลาก

## 2. ราคา ↔ การแข่งขัน แยกตามหมวด

| หมวด | จำนวน gig (supply) | ราคาเริ่มต้น median |

| เว็บ / โปรแกรมมิ่ง | 137 | **฿2,500** |
| เช่ารถ | 32 | ฿1,685 |

**ข้อสรุปสำคัญ**
- **ราคากับ supply สวนทางกัน** — ออกแบบกราฟิกเป็นหมวดที่แข่งเดือดสุด (743 gig) แต่ราคาถูกสุดๆ ระดับหนึ่ง
  (฿600) = งาน commoditised (ใครก็ทำได้ ตัดราคากัน)
- **จุดที่น่าลงทุน** (ราคาสูง + คู่แข่งน้อย) คือ **เว็บ/โปรแกรมมิ่ง** และ **สถาปัตย์/วิศวกรรม** —
  เป็นโซนที่ฟรีแลนซ์มีอำนาจตั้งราคามากที่สุด
- **รีวิวแทบไม่ช่วยแยกคุณภาพ** — มี gig แค่ ~9% ที่มีรีวิว และเกือบทุกอันได้ rating ~5.0
  → ดาว/rating เป็นสัญญาณความน่าเชื่อถือที่ "อ่อนมาก" ซึ่งเป็นเหตุผลตรงๆ ที่ต้องทำงาน anomaly ในข้อ 3

## 3. ระบบคัดกรองความน่าเชื่อถือ / ความผิดปกติ (ML)

ข้อมูลสาธารณะ **ไม่มี label ว่าอันไหนคือ fraud** เราจึงออกแบบเป็น **เครื่องมือคัดกรอง (screening)
แบบ unsupervised + rule-based** (`src/ml/anomaly.py`) — หน้าที่คือ "จัดอันดับ gig ที่ควรให้คนตรวจ
พร้อมบอกเหตุผล" ไม่ใช่ฟันธงว่าใครโกง

**กฎที่ "ยิง" บนข้อมูลจริง**
- **EXTREME_PRICE → 14 gig** ราคาเริ่มต้นผิดปกติเทียบกับหมวดตัวเอง เช่น ฿2,000,000 (ช่าง),
  ฿666,666 (ไลฟ์สไตล์), ฿650,000 (สถาปัตย์) → ราคา placeholder / ราคาปลอมชัดเจน

**กฎที่ "เงียบ" — และทำไมเรื่องนี้มีค่ามากกว่าการโชว์ตัวเลขสวยๆ ปลอมๆ**
- **กฎ fake-review: 0 ราย** — เพราะ Fastwork ผูกจำนวนรีวิวกับ order จริง (review counts consistent)
  → pattern "มีรีวิวแต่ไม่มียอดขาย" จึงมองไม่เห็นในข้อมูลสาธารณะ
- **กฎ seller-spam: 0 ราย** — การสุ่ม 2,935 จาก ~222k ทำให้ seller แต่ละคนกระจายตัว
  (เจอ seller ซ้ำสูงสุดแค่ 6 gig) → สัญญาณ "spam โพสต์ซ้ำ" ถูกทำลาย ต้องเก็บข้อมูลแบบ **targeted**
  (ดึง gig ทั้งหมดของ seller ที่สงสัย) ถึงจะตรวจเจอ
- **IsolationForest** จับความผิดปกติเชิงหลายตัวแปร แต่ "ผิดปกติ" รวมถึง **ดาวเด่น/ขายดีมาก** ด้วย
  ไม่ใช่แค่คนโกง → ผลลัพธ์จึงเป็น "triage ให้คนตรวจ" ไม่ใช่คำตัดสิน

> **ความซื่อสัตย์เชิงวิธีการคือผลลัพธ์ในตัวมันเอง:** สัญญาณไหนตรวจเจอได้ขึ้นอยู่กับวิธีสุ่มข้อมูล
> และ "ผิดปกติ ≠ ผิด"

## 4. ก้าวต่อไป

- **Targeted seller collection** — ดึง gig ทั้งหมดของ seller ที่สงสัย เพื่อเปิดใช้กฎ spam/ซ้ำ
- **ชั้น LLM explanation (optional)** — ให้ Claude แปลงแต่ละ flag เป็นคำอธิบายภาษาคนให้ผู้ตรวจอ่านง่าย
  (ตอนนี้คำอธิบายเป็นแบบ deterministic ที่ตรวจสอบย้อนได้ — auditable)
- **โมเดล supervised** — เมื่อมี label จากประวัติการตัดสินของทีม trust จริง งานนี้คือฐาน feature
  engineering ที่ต่อยอดได้ทันที

## 5. สิ่งที่สร้างขึ้น (architecture)

```
collect.py ──▶ clean.py ──▶ analyze.py / report.ipynb ──▶ dashboard/app.py
(ingest ทนทาน    (ETL +         (รายงานตลาด)              (4 tabs + filters)
 resilient,       anonymize
 resumable)       → parquet)                  └──▶ ml/anomaly.py + fraud_explore.ipynb
                                                    (คัดกรองความน่าเชื่อถือ)
```

**Data engineering ที่อยู่เบื้องหลัง (ส่วนที่หนักสุดของงาน):**
- **Ingestion ทนทาน** — robots-aware denylist, rate-limit, cache แบบ idempotent, resumable,
  ทน failure รายตัว (404/410 + per-record try/except)
- **Source-aware** — ดึงจาก `__NEXT_DATA__` ของ Next.js SPA แทน parse HTML เปราะๆ
- **ETL** — flatten JSON ซ้อน → anonymize PII (hash) → parquet (columnar) + dedup
- **Sampling** — seeded random sample กระจายทั่ว 222k URL เพื่อความ representative
- **Quality gates** — 16 tests (offline, synthetic fixtures) · ruff · GitHub Actions CI

ใช้เฉพาะข้อมูลสาธารณะแบบ aggregate/anonymized (ดู README หัวข้อ Ethics & Compliance)
